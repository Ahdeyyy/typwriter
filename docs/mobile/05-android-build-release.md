# 05 — Building, running, and releasing on Android

## Prerequisites

- Rust with the Android targets:
  ```sh
  rustup target add aarch64-linux-android armv7-linux-androideabi \
                    i686-linux-android x86_64-linux-android
  ```
- Android Studio (or at minimum the SDK + NDK + platform-tools), with
  `ANDROID_HOME` and `NDK_HOME` set.
- JDK 17+.
- `bun install` at the repo root (frontend deps).

The Android project is **already generated** under
`apps/typwriter-desktop/src-tauri/gen/android/` — you do not need
`tauri android init` on a fresh clone.

## Day-to-day commands

Run from `apps/typwriter-desktop/`:

```sh
# Dev build on a connected device/emulator with HMR:
bun tauri android dev

# Release artifacts (AAB + APK):
bun tauri android build
# APK only, specific ABI:
bun tauri android build --apk --target aarch64
```

Notes:

- `tauri android dev` serves the SvelteKit dev server to the device — the
  device/emulator must reach your machine (emulator does automatically;
  physical devices use the auto-injected local IP).
- Logs: `adb logcat` — Rust `log` output goes through tauri-plugin-log to
  logcat; filter on the app id. The in-app `get_log_file_path` command also
  works on Android (app log dir).
- Vendored OpenSSL builds from source on the first compile (the
  `openssl/vendored` target dependency) — the first Android build is slow;
  subsequent builds reuse it.

## `gen/android` — what you may and may not touch

The project is generated but **committed**, and a few files are legitimately
hand-maintained Tauri integration points. Treat it as:

- ✅ `AndroidManifest.xml` — e.g. `windowSoftInputMode="adjustResize"` (the
  mobile keyboard handling in doc 04 depends on it), permissions for
  android-fs features.
- ✅ Gradle signing config (below).
- ❌ Everything else — regenerated by Tauri version bumps; don't hand-edit
  generated Kotlin/Gradle internals, and re-check the manifest after a Tauri
  upgrade regenerates the project.

## Versioning

The Android `versionName`/`versionCode` derive from `tauri.conf.json` /
`Cargo.toml` version (`0.8.1` at time of writing). Bump the version in one
place (`Cargo.toml` + `tauri.conf.json` as the existing release flow does) —
do not hard-code versions in Gradle.

## Signing

Release builds must be signed. Standard Tauri 2 setup:

1. Create a keystore once:
   ```sh
   keytool -genkey -v -keystore typwriter-upload.jks -keyalg RSA \
           -keysize 2048 -validity 10000 -alias upload
   ```
2. Reference it from `gen/android/keystore.properties` (git-ignored):
   ```properties
   storeFile=/abs/path/typwriter-upload.jks
   storePassword=…
   keyAlias=upload
   keyPassword=…
   ```
3. The Gradle config reads that file for the `release` signing config; CI
   provides it from a secret.

Never commit the keystore or properties file.

## Updates on Android

`tauri-plugin-updater` is excluded on mobile (target-gated dependency).
Updates ship as new APKs (GitHub releases, which the landing page links) or
through a store listing if/when published. The frontend `updater` store must
remain desktop-gated; an "update available" notice on Android should link to
the release page instead of attempting an in-place update.

## CI

`.github/workflows/android.yml` builds the Android artifacts;
`nightly.yml` / `publish.yml` cover the desktop release flow. When touching
the Android workflow remember:

- Cache `~/.gradle`, the NDK, and `target/` keyed on `Cargo.lock` — the
  vendored OpenSSL build dominates cold time.
- Keystore + passwords arrive via repository secrets, written to
  `keystore.properties` at build time.
- An emulator-based smoke test (boot, open managed workspace, compile the
  onboarding doc) is the single highest-value addition — see doc 07.

## Release checklist (Android)

1. Bump version (Cargo.toml + tauri.conf.json), update the changelog.
2. `bun tauri android build` locally on a clean tree; install the APK on a
   physical device.
3. Run the manual test matrix in [07-roadmap-and-testing.md](./07-roadmap-and-testing.md)
   — at minimum: managed workspace, SAF workspace, export PDF, kill-and-restore.
4. Tag; let CI produce signed artifacts; attach to the GitHub release.
5. Verify the landing page (`typwriter-web`) picks up the release asset links.
